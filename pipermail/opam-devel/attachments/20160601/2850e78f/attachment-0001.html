<p class="MsoPlainText"><span style="color:black">Biggest feedback at the moment is that compilers-as-a-package is a fabulous innovation – I had originally expected to need to add lots of horrid Windows (and OCaml, therefore)-specific stuff to the OPAM program,
 but in fact all I've had to do is add generic features to OPAM and then handle the rest in the package itself, which is great!<o:p></o:p></span></p>
<p class="MsoPlainText"><span style="color:black"><o:p> </o:p></span></p>
<p class="MsoPlainText">> > 1. Downloading a package from several tarballs</p>
<p class="MsoPlainText">> >   I've not finished delving into the full details, but I think I'm</p>
<p class="MsoPlainText">> > correct that the url file / section can only specify a single archive,</p>
<p class="MsoPlainText">> > even if it can be made available by multiple transports? Bootstrapping</p>
<p class="MsoPlainText">> > OCaml on Windows requires two tarballs, so I was going to extend the</p>
<p class="MsoPlainText">> > url system to allow multiple tarballs to be specified (with control</p>
<p class="MsoPlainText">> > over where in the overall tree they are extracted), rather than</p>
<p class="MsoPlainText">> > hackily including the second one in files and extracting it using a</p>
<p class="MsoPlainText">> build command.</p>
<p class="MsoPlainText">> </p>
<p class="MsoPlainText">> Some other compilers, e.g. metaOCaml, have the same requirement, since</p>
<p class="MsoPlainText">> they consist of the official OCaml tarball plus a huge patch, so I have</p>
<p class="MsoPlainText">> added a way to handle this.</p>
<p class="MsoPlainText">> </p>
<p class="MsoPlainText">> The `url` section or file still is a single source, and the commands are</p>
<p class="MsoPlainText">> executed from its root (after extraction if necessary). But there is now</p>
<p class="MsoPlainText">> also an opam field `extra-sources:` that you can use to download more</p>
<p class="MsoPlainText">> files, unextracted, to the build directory. The format is:</p>
<p class="MsoPlainText">> </p>
<p class="MsoPlainText">>     extra-sources: [ [ "URL" { "target-filename" } "MD5" ] ... ]</p>
<p class="MsoPlainText">> </p>
<p class="MsoPlainText">> The "target-filename" part being optional. See for example</p>
<p class="MsoPlainText">> <a href="http://opam.ocaml.org/2.0~dev/packages/ocaml/ocaml.4.01.0+BER/opam">
<span style="color:windowtext;text-decoration:none">http://opam.ocaml.org/2.0~dev/packages/ocaml/ocaml.4.01.0+BER/opam</span></a></p>
<p class="MsoPlainText"><span style="color:black"><o:p> </o:p></span></p>
<p class="MsoPlainText"><span style="color:black">Ta, I came across it subsequently too! I've extended it (or rather, will shortly extend it), to allow it to refer to the URL of another package, as the ocaml package needs to pick up the sources of the flexdll
 package installed just before it in order to bootstrap correctly.<o:p></o:p></span></p>
<p class="MsoPlainText"><o:p> </o:p></p>
<p class="MsoPlainText">> > 2. Specifying package variables at opam switch install</p>
<p class="MsoPlainText">> >   I've extended the ocaml package variables to include three new ones:</p>
<p class="MsoPlainText">> > ocaml-arch (which is exactly the architecture determined by OCaml's</p>
<p class="MsoPlainText">> > configure script), ocaml-cc (which is "cl" for a Microsoft C compiler</p>
<p class="MsoPlainText">> > based OCaml or "cc" for the rest of the world), and ocaml-libc (which is</p>
<p class="MsoPlainText">> "msvc"</p>
<p class="MsoPlainText">> > for the four native Windows OCaml ports and "libc" for the rest of the</p>
<p class="MsoPlainText">> > world).</p>
<p class="MsoPlainText">> ></p>
<p class="MsoPlainText">> >   At OPAM switch time, one selects which Windows port you get by</p>
<p class="MsoPlainText">> > specifying these three variables. So, for example, mingw64 is</p>
<p class="MsoPlainText">> > ocaml-arch=amd64, ocaml-cc=cc, ocaml-libc=msvc; msvc32 is</p>
<p class="MsoPlainText">> > ocaml-arch=i386, ocaml-cc=cl, ocaml-libc=msvc; cygwin64 is</p>
<p class="MsoPlainText">> > ocaml-arch=amd64, ocaml-cc=cc, ocaml-libc=libc.</p>
<p class="MsoPlainText">> ></p>
<p class="MsoPlainText">> >   My question is how to specify these - the concept of "architecture",</p>
<p class="MsoPlainText">> > "C compiler type" and "C runtime library" is generic enough that it</p>
<p class="MsoPlainText">> > doesn't feel too bad to have opam switch --cc=cl --arch=amd64 4.03.0</p>
<p class="MsoPlainText">> > and have the package initialise ocaml-arch, etc. based on those</p>
<p class="MsoPlainText">> > globally available variables. However, is there any plan (or existing</p>
<p class="MsoPlainText">> > feature) for a more generic way of specifying package input variables?</p>
<p class="MsoPlainText">> > (discussed in <a href="https://github.com/ocaml/opam/issues/2247">
<span style="color:windowtext;text-decoration:none">https://github.com/ocaml/opam/issues/2247</span></a>).</p>
<p class="MsoPlainText">> </p>
<p class="MsoPlainText">> a) for the _output_ variables, you can define them in your `PKG.config`</p>
<p class="MsoPlainText">> variable (automatically installed by opam if present in your build dir).</p>
<p class="MsoPlainText">> Note that, at the moment, variables defined by the compiler package get</p>
<p class="MsoPlainText">> exported as global variables (so you can access e.g. `ocaml-version`</p>
<p class="MsoPlainText">> anywhere), but there are limitations due to this and it is likely to</p>
<p class="MsoPlainText">> disappear from the final release (see</p>
<p class="MsoPlainText">> <a href="https://github.com/ocaml/opam/issues/2537"><span style="color:windowtext;text-decoration:none">https://github.com/ocaml/opam/issues/2537</span></a> for details). You can always</p>
<p class="MsoPlainText">> access them through `ocaml:VAR` inside the script parts though.</p>
<p class="MsoPlainText">> </p>
<p class="MsoPlainText">> b) nothing yet for _input_ variables, but #2247 is still something I need</p>
<p class="MsoPlainText">> to work on. You can define global variables through `opam config set`,</p>
<p class="MsoPlainText">> though, so an option in the meantime could be to use `opam switch install</p>
<p class="MsoPlainText">> --empty`, set the variables, then install the compiler (the option to</p>
<p class="MsoPlainText">> install and mark as compiler is missing at the moment, though, for that</p>
<p class="MsoPlainText">> you'll have to edit</p>
<p class="MsoPlainText">> `SWITCH/.opam-switch/switch-state`)</p>
<p class="MsoPlainText"><span style="color:black"><o:p> </o:p></span></p>
<p class="MsoPlainText"><span style="color:black">I've gone with the command line options for now (with an error check to ensure that they're only given when installing a switch). At the moment, they define three global variables switch-cc, switch-libc and
 switch-arch which are only available during the switch installation, so if they want to be kept, the compiler package has to capture them in its own .config file. ocaml.system remains a horrible hack under this mechanism (the script identifies that it's in
 a switch named "system" and uses that fact to probe switch-cc, switch-libc, and switch-arch...)<o:p></o:p></span></p>
<p class="MsoPlainText"><span style="color:black"><o:p> </o:p></span></p>
<p class="MsoPlainText"><span style="color:black">It's not particularly pretty, but it does seem to work in the absence of something more general for now. In particular, I've restructured the base packages and added a package called compiler whose purpose is
 to store variables (in particular, compiler:o, compiler:so and compiler:a giving important file extensions for object files, etc., which are necessary for writing generic package specifications). The only bug I had to fix with that so far is to thread the
 config files correctly through parallel_apply (before they were only re-loaded at the end of the installation), but there is a comment in the code that you already aware of problems there!<o:p></o:p></span></p>
<p class="MsoPlainText"><o:p> </o:p></p>
<p class="MsoPlainText">> > 3. Upgrading dependent packages for the root</p>
<p class="MsoPlainText">> >   Any Windows OCaml installation will depend upon FlexDLL objects</p>
<p class="MsoPlainText">> > compiled with the specific C compiler combination. Although in</p>
<p class="MsoPlainText">> > principle these could be shared between multiple switches (the object</p>
<p class="MsoPlainText">> > files vary according ocaml-cc, ocaml-libc and ocaml-arch; not</p>
<p class="MsoPlainText">> > ocaml-version), it makes sense to compile these two object files</p>
<p class="MsoPlainText">> > per-switch, rather than introduce yet another installation layer.</p>
<p class="MsoPlainText">> ></p>
<p class="MsoPlainText">> >   This means that in addition to the base packages, the Windows OCaml</p>
<p class="MsoPlainText">> > package depends on a flexdll package. Is there any philosophical</p>
<p class="MsoPlainText">> > reason for that not to be part of opam upgrade? Obviously, one doesn't</p>
<p class="MsoPlainText">> > want to be trying to upgrade the ocaml package, but why lock its</p>
<p class="MsoPlainText">> > dependencies at a given version? It would be quite nice if a new</p>
<p class="MsoPlainText">> > version of flexdll is released to be able to have opam upgrade notice</p>
<p class="MsoPlainText">> > that flexdll can be upgraded, determine that the root package must</p>
<p class="MsoPlainText">> > therefore be recompiled and so automatically recompile the entire</p>
<p class="MsoPlainText">> switch.</p>
<p class="MsoPlainText">> ></p>
<p class="MsoPlainText">> >   Is it just for philosophical reasons that the dependencies of the</p>
<p class="MsoPlainText">> > root package are not considered for upgrade or is there a technical</p>
<p class="MsoPlainText">> > consideration?</p>
<p class="MsoPlainText">> </p>
<p class="MsoPlainText">> Yes, just philosophical, there is no constraint here besides the arbitrary</p>
<p class="MsoPlainText">> "I don't want to recompile the compiler, ever".</p>
<p class="MsoPlainText"><span style="color:black"><o:p> </o:p></span></p>

</span>I've altered opam switch so that only the root package is the compiler package – so only ocaml.version is in the compiler section of switch-state.<span style="color:black"> Have a look at this transcript:<o:p></o:p></span></p>
