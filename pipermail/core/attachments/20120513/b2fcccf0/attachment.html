<p>The issue is that, since the repo is a subset of our internal tree, some information is lost when merges of internal revisions are done in the bitbucket repo. It's safest to not pull in merge nodes at all, but to be precise, as long as the incoming merge nodes are descending from a single revision that has already been pulled in, it's okay. </p>

<p>In this case, if all of the incoming revisions are descending from Ron's (or my) changers, merge nodes are safe to pull. </p>
